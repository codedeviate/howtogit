# pr

Manage the full lifecycle of GitHub pull requests from the command line:
create, inspect, review, merge, and close — without leaving your terminal.

## Mental model

A pull request (PR) is GitHub's unit of code review. It asks: "please merge
the commits on my branch into your branch." `gh pr` is a direct API client
for that concept. Every subcommand targets a specific PR, identified by its
number, URL, or head branch name — `gh pr` resolves them all to the same
thing.

When you run a command against "the current branch" and omit an explicit PR
identifier, `gh pr` looks up which open PR has your branch as its head. This
means you can `gh pr view`, `gh pr checks`, and `gh pr merge` without ever
typing a number — as long as your branch has a PR and you are on that branch.

The subcommands fall into two groups:

- **General commands** (`create`, `list`, `status`) — work across many PRs or
  create a new one.
- **Targeted commands** (`view`, `checkout`, `checks`, `review`, `merge`,
  `edit`, `diff`, `comment`, `ready`, `close`, `reopen`, `revert`,
  `update-branch`, `lock`, `unlock`) — operate on a single PR.

## Synopsis

```text
gh pr create       [flags]
gh pr list         [flags]
gh pr status       [flags]

gh pr view         [<number> | <url> | <branch>] [flags]
gh pr checkout     [<number> | <url> | <branch>] [flags]
gh pr checks       [<number> | <url> | <branch>] [flags]
gh pr diff         [<number> | <url> | <branch>] [flags]
gh pr review       [<number> | <url> | <branch>] [flags]
gh pr comment      [<number> | <url> | <branch>] [flags]
gh pr edit         [<number> | <url> | <branch>] [flags]
gh pr merge        [<number> | <url> | <branch>] [flags]
gh pr ready        [<number> | <url> | <branch>] [flags]
gh pr update-branch [<number> | <url> | <branch>] [flags]
gh pr revert       {<number> | <url> | <branch>} [flags]
gh pr close        {<number> | <url> | <branch>} [flags]
gh pr reopen       {<number> | <url> | <branch>} [flags]
gh pr lock         {<number> | <url> | <branch>} [flags]
gh pr unlock       {<number> | <url> | <branch>} [flags]
```

All targeted subcommands accept `-R OWNER/REPO` to operate on a different
repository than the one detected from the current directory.

## Everyday usage

### Opening a new pull request

The fastest path when your branch commits already describe the change:

```sh
gh pr create --fill
```

`--fill` uses the commits' titles and bodies for the PR title and description.
Omit it to get an interactive prompt, or go fully non-interactive:

```sh
gh pr create --title "Fix login redirect loop" --body "Closes #42"
```

Open as a draft when the work is not yet ready for review:

```sh
gh pr create --draft --title "WIP: refactor auth middleware"
```

### Listing open pull requests

```sh
gh pr list
```

Narrow by author, label, or state:

```sh
gh pr list --author "@me"
gh pr list --label "needs-review" --state open
gh pr list --state merged --limit 10
```

### Checking the status of your own PRs

```sh
gh pr status
```

This shows PRs you authored, PRs assigned to you, and PRs that mention you.

### Viewing a PR

```sh
gh pr view          # PR for the current branch
gh pr view 123
gh pr view --web    # open in browser
```

Add `--comments` to see the discussion thread:

```sh
gh pr view 123 --comments
```

### Checking CI status

```sh
gh pr checks        # for the current branch's PR
gh pr checks 123
```

Watch checks live until they all finish:

```sh
gh pr checks --watch
gh pr checks --watch --fail-fast   # stop watching on the first failure
```

### Reviewing a PR

Approve the PR for the current branch:

```sh
gh pr review --approve
```

Request changes with a message:

```sh
gh pr review 123 --request-changes --body "Please add tests for the edge case."
```

Leave a comment without approving or blocking:

```sh
gh pr review 123 --comment --body "Looks good, left a nit."
```

### Merging

Merge with a merge commit:

```sh
gh pr merge 123 --merge
```

Squash all commits into one:

```sh
gh pr merge 123 --squash --delete-branch
```

Rebase the commits onto the base branch:

```sh
gh pr merge 123 --rebase
```

Enable auto-merge (merges automatically once all required checks pass):

```sh
gh pr merge 123 --auto --squash
```

### Checking out a PR locally

```sh
gh pr checkout 123
```

Override the local branch name:

```sh
gh pr checkout 123 --branch fix-login-local
```

### Commenting on a PR

```sh
gh pr comment 123 --body "LGTM"
```

### Editing a PR after creation

Change the title and add a label:

```sh
gh pr edit 123 --title "Fix: login redirect loop" --add-label "bug"
```

Request an additional reviewer:

```sh
gh pr edit 123 --add-reviewer octocat
```

### Marking a draft PR as ready

```sh
gh pr ready 123
```

Convert a ready PR back to draft:

```sh
gh pr ready 123 --undo
```

### Updating a PR branch that has fallen behind

Merge the latest base branch into the PR branch:

```sh
gh pr update-branch 123
```

Rebase instead:

```sh
gh pr update-branch 123 --rebase
```

### Closing and reopening

```sh
gh pr close 123 --comment "Superseded by #200"
gh pr reopen 123 --comment "Reviving after further discussion"
```

### Reverting a merged PR

```sh
gh pr revert 123
```

This creates a new PR that undoes the changes introduced by PR 123.

### Viewing the diff

```sh
gh pr diff         # diff for current branch's PR
gh pr diff 123
gh pr diff 123 --name-only
gh pr diff 123 --exclude '*.lock'
```

## Key options

### create

| Flag | What it does | When to use it |
|------|--------------|----------------|
| `-f` / `--fill` | Populate title and body from commit messages | The commits already explain the change |
| `--fill-first` | Use only the first commit for title and body | A single-commit PR |
| `--fill-verbose` | Use full commit message bodies in the description | When commit bodies contain important context |
| `-d` / `--draft` | Open as a draft | Work in progress, not yet ready for review |
| `-t` / `--title` | Set the PR title | Non-interactive workflows |
| `-b` / `--body` | Set the PR body | Non-interactive workflows |
| `-F` / `--body-file` | Read the body from a file | Long descriptions prepared in an editor |
| `-B` / `--base` | Target branch to merge into | When the default branch is not the right target |
| `-r` / `--reviewer` | Request review from a handle or team | Assign reviewers at creation time |
| `-a` / `--assignee` | Assign people; `@me` for yourself | Assign at creation time |
| `-l` / `--label` | Add labels by name | Categorise the PR immediately |
| `--no-maintainer-edit` | Prevent maintainers from pushing to the head branch | Forks where you want full control |
| `-w` / `--web` | Open the browser to finish creating | Prefer the web form |
| `--dry-run` | Print details instead of creating the PR (may still push git changes) | Validate flags before running for real |
| `-T` / `--template` | Start from a body template file | Enforce a consistent PR format |

### list

| Flag | What it does | When to use it |
|------|--------------|----------------|
| `-s` / `--state` | `open`, `closed`, `merged`, or `all` | Review historical PRs |
| `-A` / `--author` | Filter by author login; `@me` for yourself | Find your own PRs |
| `-a` / `--assignee` | Filter by assignee | Your review queue |
| `-l` / `--label` | Filter by one or more labels | Triage by category |
| `-B` / `--base` | Filter by target branch | PRs targeting a release branch |
| `-S` / `--search` | GitHub search query | Complex filters (e.g., `review:required`) |
| `-L` / `--limit` | Maximum results (default 30) | Wider history scans |
| `--json` | Output as JSON with named fields | Scripting and automation |
| `-q` / `--jq` | Filter JSON output with a jq expression | Extract specific fields |
| `-w` / `--web` | Open list in browser | Browse interactively |

### checks

| Flag | What it does | When to use it |
|------|--------------|----------------|
| `--watch` | Poll until all checks complete | Block a script until CI finishes |
| `--fail-fast` | Stop watching on the first failure (requires `--watch`) | Fail scripts early |
| `-i` / `--interval` | Refresh interval in seconds (default 10) | Reduce API calls on slow CI |
| `--required` | Show only required checks | Cut noise when many optional checks exist |
| `-w` / `--web` | Open the checks page in a browser | Inspect individual job logs |

### merge

| Flag | What it does | When to use it |
|------|--------------|----------------|
| `-m` / `--merge` | Merge commit | Preserve full history |
| `-s` / `--squash` | Squash into one commit | Clean up a noisy branch |
| `-r` / `--rebase` | Rebase onto base branch | Linear history without merge commits |
| `-d` / `--delete-branch` | Delete the local and remote branch after merge | Keep the repo tidy |
| `--auto` | Merge automatically once requirements are met | Merge queues and required checks |
| `--disable-auto` | Cancel a previously set auto-merge | Changed your mind |
| `--admin` | Bypass branch-protection rules | Emergency fix with administrator access |
| `--match-head-commit` | Only merge if the PR head matches a specific SHA | Guard against last-second pushes |

### review

| Flag | What it does | When to use it |
|------|--------------|----------------|
| `-a` / `--approve` | Approve the PR | Standard approval |
| `-r` / `--request-changes` | Request changes | Block merge until addressed |
| `-c` / `--comment` | Comment without approving or blocking | Leaving notes |
| `-b` / `--body` | Inline review message | Required when requesting changes |
| `-F` / `--body-file` | Read the review body from a file | Long prepared reviews |

### edit

| Flag | What it does | When to use it |
|------|--------------|----------------|
| `-t` / `--title` | Change the PR title | Correct a typo after creation |
| `-b` / `--body` | Replace the PR body | Update the description |
| `-B` / `--base` | Change the target branch | Retarget to a different branch |
| `--add-reviewer` / `--remove-reviewer` | Manage reviewers; `@copilot` for Copilot review | Re-request review after updates |
| `--add-assignee` / `--remove-assignee` | Manage assignees; `@me` and `@copilot` supported | Reassign during review |
| `--add-label` / `--remove-label` | Manage labels | Relabel without the web UI |
| `--add-project` / `--remove-project` | Manage project associations | Requires `project` scope; see *auth* |

### diff

| Flag | What it does | When to use it |
|------|--------------|----------------|
| `--name-only` | List changed filenames only | Quick impact scan |
| `-e` / `--exclude` | Exclude files matching a glob pattern | Filter out generated files or lock files |
| `--patch` | Output in standard patch format | Piping into `git apply` or other tools |
| `--color` | `always`, `never`, or `auto` | Scripting without colour codes |

## Best practices

**Use `--fill` for most PRs.** Writing good commit messages pays off
immediately: `gh pr create --fill` turns them into the PR description with no
extra effort.

**Open drafts early.** A draft PR (`--draft`) signals work in progress,
triggers CI, and starts the conversation before the code is final. Convert it
with `gh pr ready` when you are done.

**Prefer `--auto --squash` (or `--rebase`) over manual timing.** When a
repository enforces required checks, `gh pr merge --auto` queues the merge and
walks away. You do not need to poll and re-run the merge command after CI
passes.

**Delete the branch on merge.** Pass `--delete-branch` to `gh pr merge` to
clean up both the remote and local tracking branch in one step. A growing list
of stale branches is one of the most common sources of confusion on shared
repositories.

**Use `--search` for non-trivial filtering.** The `--author`, `--label`, and
`--base` flags handle common cases, but `gh pr list --search` accepts the full
GitHub search syntax:

```sh
gh pr list --search "review:required is:open" --state open
gh pr list --search "base:release/2.0 is:merged"
```

**Let `gh pr checks --watch` gate automation.** Instead of polling in a shell
loop, let `gh` do it:

```sh
gh pr checks --watch && gh pr merge --squash --delete-branch
```

Exit code 0 means all checks passed; non-zero means at least one failed.

**Request Copilot review when available.** On supported plans,
`--add-reviewer "@copilot"` requests an automated code review from GitHub
Copilot — useful as a first pass before human reviewers see the change:

```sh
gh pr edit 123 --add-reviewer "@copilot"
```

## Pitfalls & gotchas

**`gh pr create` will offer to fork and push if your branch is local-only.**
If you are working on a fork already, double-check the prompt before
confirming — you may end up with a fork of a fork. Use `--head OWNER:BRANCH`
to be explicit about which remote branch is the head.

**`--fill` is overridden by explicit `--title` / `--body`.**
If you specify both `--fill` and `--title`, the explicit title wins. This is
intentional, but it surprises people who expect `--fill` to be a base that
explicit flags extend rather than replace.

**Auto-merge and merge queues interact differently.**
If the base branch has a merge queue enabled, `gh pr merge --auto` adds the PR
to the queue rather than merging directly. To bypass the queue you need
`--admin`, which requires administrator privileges.

**`gh pr checkout` creates a local branch that tracks the remote.**
If the PR author later force-pushes their branch, your local branch will be
behind. Use `--force` to reset it to the current state:

```sh
gh pr checkout 123 --force
```

**`gh pr review --request-changes` without `--body` is rejected.**
GitHub requires a comment body when requesting changes. Always include `-b`:

```sh
gh pr review 123 --request-changes --body "Please add error handling."
```

**`edit --add-project` requires the `project` scope.**
If you get a scope error, add it once and continue — no logout required:

```sh
gh auth refresh --scopes project
```

See *auth* for details on managing scopes.

**`close` does not delete the branch by default.**
Unlike `merge`, closing a PR without `--delete-branch` leaves the remote
branch intact. Add the flag explicitly when you want cleanup:

```sh
gh pr close 123 --delete-branch
```

**`--state merged` in `gh pr list` is distinct from `--state closed`.**
Merged PRs are a subset of closed PRs. Using `--state closed` returns both
merged and abandoned PRs. Use `--state merged` when you are searching for
merged work specifically.

## Worked examples

### Opening a feature PR with reviewers, labels, and a milestone

You have finished a feature branch `feat/dark-mode`:

```sh
# Make sure the branch is pushed
git push -u origin feat/dark-mode

# Create the PR, requesting two reviewers and tagging the sprint milestone
gh pr create \
  --fill \
  --reviewer octocat,myorg/frontend-team \
  --label "enhancement" \
  --milestone "Sprint 24"
```

```text
https://github.com/myorg/myapp/pull/87
```

A colleague pushes follow-up commits, so CI re-runs. Watch from the terminal:

```sh
gh pr checks 87 --watch
```

```text
All checks were successful
0 failing, 0 pending, and 6 passing checks
```

Approve and merge when ready:

```sh
gh pr review 87 --approve
gh pr merge 87 --squash --delete-branch
```

### Reviewing a contributor's PR from a fork

A contributor opened PR 201 from their fork. Check out the branch to run it
locally:

```sh
gh pr checkout 201
```

```text
remote: Enumerating objects: 12, done.
From https://github.com/myorg/myapp
 * [new ref]         refs/pull/201/head -> patch-1
Switched to branch 'patch-1'
```

Read the diff for a quick overview before diving in:

```sh
gh pr diff 201 --name-only
```

```text
src/auth/middleware.ts
tests/auth/middleware.test.ts
```

Submit a structured review with a comment:

```sh
gh pr review 201 --comment \
  --body "Tested locally — the middleware behaves correctly. One style nit: prefer \`const\` over \`let\` on line 42."
```

If a nit is blocking, request changes:

```sh
gh pr review 201 --request-changes \
  --body "Please rename the variable to match the project convention (camelCase)."
```

After the contributor pushes a fix, re-request your review and approve:

```sh
gh pr edit 201 --add-reviewer "@me"
gh pr review 201 --approve
```

### Handling a stale PR branch

PR 150 was opened two weeks ago. The base branch has moved on and GitHub shows
a conflict warning. Update the PR branch without leaving the terminal:

```sh
gh pr update-branch 150
```

If the team prefers a linear history:

```sh
gh pr update-branch 150 --rebase
```

Then re-check CI:

```sh
gh pr checks 150 --watch
```

### Reverting a bad merge

PR 99 introduced a regression. Create a revert PR immediately:

```sh
gh pr revert 99 --title "Revert: fix regression from PR 99"
```

GitHub creates a new PR with an inverse diff of PR 99. Merge it through the
normal review process.

### Scripting: find all open PRs you need to review

Extract PR numbers, author logins, and titles using JSON output and jq:

```sh
gh pr list \
  --search "review-requested:@me is:open" \
  --json number,title,author \
  --jq '.[] | "\(.number)\t\(.author.login)\t\(.title)"'
```

```text
204	contributor-a	Add dark mode
198	contributor-b	Fix login redirect
```

## Recovery

**PR created against the wrong base branch.** Edit it without closing and
reopening:

```sh
gh pr edit 123 --base correct-base-branch
```

**Draft PR accidentally published.** Convert back to draft:

```sh
gh pr ready 123 --undo
```

**Merge failed due to conflicts.** Update the branch, resolve locally, push,
then retry:

```sh
gh pr update-branch 123        # or --rebase
# resolve conflicts in your editor, then:
git add .
git commit
git push
gh pr merge 123 --squash --delete-branch
```

**Merged the wrong PR by mistake.** Revert it:

```sh
gh pr revert 123
```

This opens a new PR with the inverse diff — review and merge that PR to undo
the original change.

**`gh pr checkout` put you on a detached HEAD.** Run the command without
`--detach`, or create a named branch from where you are:

```sh
gh pr checkout 123              # creates a named local branch by default
```

For broader undo patterns, see *Getting out of jams*.
