# completion

Generate shell completion scripts so your shell can tab-complete `gh`
commands, subcommands, flags, and arguments.

## Mental model

When you press Tab after typing `gh pr`, your shell does not ask GitHub for
a list of subcommands. Instead, it consults a completion script — a chunk of
shell code that was loaded at startup time, describing every command, flag,
and accepted value. `gh completion` outputs that script for your shell.

The script is static: it encodes the structure of `gh` at the time the CLI
was built. It does not query the network. Once loaded, it makes the entire
`gh` command tree navigable with Tab without any runtime cost.

Most package managers install the completion script automatically. If you
installed `gh` with Homebrew on macOS, completions are already wired up
through Homebrew's own completion infrastructure. `gh completion` matters
most when you installed the binary manually, upgraded to a new version whose
new subcommands are not yet reflected in the system-installed script, or need
to set up completions in a non-standard shell profile.

## Synopsis

```text
gh completion -s <shell>
```

The `-s` flag is required. There are no subcommands.

## Everyday usage

Print a completion script for the current session (useful for testing):

```sh
gh completion -s zsh
```

Install completions for zsh permanently:

```sh
gh completion -s zsh > /usr/local/share/zsh/site-functions/_gh
```

Install completions for bash (requires `bash-completion` to be installed):

```sh
# Add to ~/.bash_profile
eval "$(gh completion -s bash)"
```

Install completions for fish:

```sh
gh completion -s fish > ~/.config/fish/completions/gh.fish
```

Install completions for PowerShell. Open your profile with:

```sh
notepad $profile
```

Then add and save:

```text
Invoke-Expression -Command $(gh completion -s powershell | Out-String)
```

After writing or sourcing any of these files, restart your shell before
testing Tab completion.

## Key options

| Flag | What it does | When to use it |
|------|--------------|----------------|
| `-s` / `--shell` | Shell type: `bash`, `zsh`, `fish`, or `powershell` | Required on every invocation |

There are no other flags beyond `--help`.

## Best practices

**Write the script to a file rather than eval-ing it on every startup.**
Evaling a script at shell startup — `eval "$(gh completion -s zsh)"` — works
but re-runs the script on every new shell. For zsh and fish, writing once to
a file in `$fpath` (zsh) or `~/.config/fish/completions/` (fish) loads faster
and survives `gh` upgrades as-is until you explicitly refresh it.

**Re-run after upgrading `gh`.** The script encodes the subcommands and flags
of the exact version that generated it. After a major upgrade that adds new
commands, overwrite the completion file so Tab discovers the new additions:

```sh
gh completion -s zsh > /usr/local/share/zsh/site-functions/_gh
exec zsh   # reload the shell
```

**Let Homebrew manage completions when possible.** If you installed `gh` via
Homebrew, follow the Homebrew shell-completion setup instead of running
`gh completion` directly. Homebrew refreshes the script automatically on
upgrades. See `https://docs.brew.sh/Shell-Completion` for setup instructions.

**Verify `$fpath` contains your target directory before writing.** For zsh,
the file must land in a directory that appears in `$fpath`. Print the list to
confirm:

```sh
print -l $fpath
```

If `/usr/local/share/zsh/site-functions` is absent, pick a directory that is
present, or add a custom one and ensure `compinit` runs after it:

```sh
fpath=(~/.zsh/completions $fpath)
autoload -U compinit && compinit
```

## Pitfalls & gotchas

**Completions not working after installation.** The most common cause is
forgetting to restart the shell (or run `exec zsh` / `exec bash`). The
completion script is sourced at shell startup; an already-running shell does
not pick up the new file automatically. If a restart does not help, check
that `compinit` is called in `~/.zshrc` *after* the `$fpath` addition:

```sh
autoload -U compinit
compinit -i
```

**zsh `compinit: insecure directories` warning.** The `-i` flag to
`compinit` suppresses this warning by ignoring insecure directories. Without
it, zsh may refuse to load completions from world-writable directories.
Alternatively, fix the directory permissions with `chmod go-w`.

**The generated script does not include `gh` extensions.** Completion scripts
cover only built-in commands. Tab completion for commands added through
`gh extension install` (see *extension*) is not generated by `gh completion`
— each extension would need to provide its own completion logic.

**Homebrew completions and a manual file conflict.** If Homebrew is managing
a completion file and you also write one manually to the same path, they will
overwrite each other on the next `brew upgrade`. Choose one approach and stick
to it. A safe manual location outside Homebrew's reach is `~/.zsh/completions/`.

**PowerShell profile directory may not exist.** The `notepad $profile` step
will fail if the parent directory has not been created yet. Create it first:

```powershell
mkdir -Path (Split-Path -Parent $profile) -ErrorAction SilentlyContinue
```

## Worked examples

### Setting up zsh completions from scratch

Check whether a suitable directory exists in `$fpath`:

```sh
print -l $fpath | grep site-functions
```

```text
/usr/local/share/zsh/site-functions
```

Write the script there:

```sh
gh completion -s zsh > /usr/local/share/zsh/site-functions/_gh
```

Ensure `~/.zshrc` contains the required lines (add them if missing):

```sh
autoload -U compinit
compinit -i
```

Reload the shell and test:

```sh
exec zsh
gh pr <Tab>
```

```text
checkout  -- Check out a pull request in git
close     -- Close a pull request
comment   -- Add a comment to a pull request
...
```

### Refreshing completions after a gh upgrade

After upgrading `gh` the new `gh attestation` subcommands do not Tab-complete
because the on-disk script was generated by the old binary:

```sh
# Overwrite the stale file with the freshly-built script
gh completion -s zsh > /usr/local/share/zsh/site-functions/_gh

# Reload
exec zsh

# Verify the new subcommand appears
gh attestation <Tab>
```

```text
download  -- Download an artifact attestation
inspect   -- Inspect a sigstore bundle used for attestation verification
...
```

### Adding fish completions

```sh
# Create the directory if it does not exist
mkdir -p ~/.config/fish/completions

# Write the script
gh completion -s fish > ~/.config/fish/completions/gh.fish
```

Open a new fish session (or run `exec fish`) and test:

```sh
gh repo <Tab>
```

```text
archive   clone   create   delete   deploy-key   edit   fork ...
```

## Recovery

If Tab completion stops working after a `gh` upgrade, regenerate the script
and reload the shell:

```sh
gh completion -s zsh > /usr/local/share/zsh/site-functions/_gh
exec zsh
```

If the file was written to the wrong location, identify where the shell looks
for completion scripts:

```sh
# zsh: list every directory searched for completion files
print -l $fpath

# bash: check the compat directory
echo $BASH_COMPLETION_COMPAT_DIR
```

Write the script to a directory that appears in that list, then reload.

For Homebrew-managed installations, let Homebrew regenerate the script by
running `brew reinstall gh` rather than writing the file manually.

## See also

- *auth* — authenticate `gh` before any command can be tab-completed and
  executed.
- *alias* — custom aliases defined with `gh alias set` are not covered by
  the generated completion script.
- *extension* — third-party extensions installed with `gh extension install`
  require their own completion setup outside of `gh completion`.
- *config* — `gh config set` controls shell and editor preferences that
  complement the completion setup.
