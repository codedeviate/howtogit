# rm

Remove files from the index, or from both the working tree and the index, and
stage that removal for the next commit.

## Mental model

When you delete a file with your operating system's `rm` command, the working
tree copy disappears but Git does not know about it yet. The index still lists
the file, so `git status` shows it as "deleted" (unstaged). You would then
have to run `git add` on the deletion to stage it, or `git commit -a` to pick
it up automatically.

`git rm` collapses those two steps into one: it removes the file from the
working tree *and* stages the removal, so your next `git commit` will record
it as gone.

```text
File tracked by Git
      │
      ├─ os rm ──> file gone from disk, deletion NOT staged
      │              └─ git add <path>  ──> deletion staged ──> commit
      │
      └─ git rm ──> file gone from disk, deletion staged   ──> commit
```

The `--cached` flag breaks the symmetry: it removes the file from the index
only, leaving the working tree copy intact. This is the right move when you
accidentally committed a file that should have been in `.gitignore`.

Git refuses to remove a file whose index entry differs from either the last
commit or the current working tree, because that would discard unsaved work.
Pass `-f` to override that safety check when you are sure.

## Synopsis

```text
git rm [-f | --force] [-n] [-r] [--cached] [--ignore-unmatch]
       [--quiet] [--pathspec-from-file=<file> [--pathspec-file-nul]]
       [--] [<pathspec>...]
```

## Everyday usage

Remove a file from the working tree and stage the deletion:

```sh
git rm src/legacy.js
git commit -m "Remove legacy.js"
```

Remove a file from Git's tracking without deleting it from disk (the most
common use after updating `.gitignore`):

```sh
git rm --cached config/secrets.yml
echo "config/secrets.yml" >> .gitignore
git commit -m "Stop tracking secrets.yml"
```

Remove an entire directory recursively:

```sh
git rm -r build/
git commit -m "Remove generated build artifacts"
```

Preview what would be removed without touching anything:

```sh
git rm -n -r dist/
```

Remove files whose names might look like options:

```sh
git rm -- -oddly-named-file.txt
```

## Key options

| Option | What it does | When to use it |
|--------|--------------|----------------|
| `--cached` | Remove from the index only; leave the working tree file alone | Untrack a file without deleting it (e.g. after updating `.gitignore`) |
| `-f`, `--force` | Override the up-to-date check | Remove a file with uncommitted local changes you are sure you do not need |
| `-n`, `--dry-run` | Show what would be removed without removing anything | Verify a glob or directory removal before committing to it |
| `-r` | Allow recursive removal of a directory | Required whenever a directory name is given instead of individual files |
| `--ignore-unmatch` | Exit with status 0 even when no files match the pathspec | Scripts that should not fail when the file is already gone |
| `-q`, `--quiet` | Suppress the per-file `rm` lines on stdout | Batch operations where the output would be noisy |
| `--pathspec-from-file=<file>` | Read pathspecs from a file instead of the command line | Large lists of files, or paths generated by another command |
| `--pathspec-file-nul` | Use NUL as the separator in `--pathspec-from-file` | Filenames that contain newlines or spaces |
| `--sparse` | Allow updating index entries outside the sparse-checkout cone | Sparse-checkout workflows |

## Best practices

**Prefer `git rm` over `rm` + `git add` for tracked deletions.** The
two-step approach works, but `git rm` is shorter, harder to forget halfway,
and makes your intent immediately visible in the shell history.

**Use `--cached` whenever you update `.gitignore`.** If a file is already
tracked, adding its path to `.gitignore` does nothing — Git continues to track
changes to it. You must untrack it with `git rm --cached <path>` first. The
file stays on disk; only Git's awareness of it is removed.

**Always pass `-n` before a broad glob or directory removal.** Globbing in
Git can behave differently from the shell (see *Pitfalls & gotchas*). A quick
dry run costs nothing and prevents accidentally removing files you meant to
keep.

**Commit the removal promptly.** `git rm` stages a deletion. Until you
commit, the removal is sitting in the index as an uncommitted change. Other
commands such as `git checkout` or `git reset` can disturb it. Stage the
removal and commit in the same logical step.

**Use `--ignore-unmatch` in scripts and cleanup routines.** When a cleanup
script runs `git rm` on paths that may or may not exist, the default non-zero
exit on "no match" will abort the script. `--ignore-unmatch` makes the
operation idempotent.

## Pitfalls & gotchas

**Git globbing crosses directory boundaries; shell globbing does not.** When
you quote the glob (`git rm 'src/*.js'`), Git expands it and will match files
in any subdirectory of `src`. When you leave it unquoted (`git rm src/*.js`),
the shell expands it first and matches only the top-level `src/` entries. The
manual example `git rm 'd*'` would also match a sibling directory `d2/` — a
surprise if you only intended `d/`. Prefer explicit paths or use `-n` to
preview.

**`git rm` will not remove only from the working tree.** There is no flag to
delete a file from disk while leaving the index entry untouched. Use your
system's `rm` for that. `git rm` always touches the index.

**Removing a file with uncommitted local changes fails by default.** Git
compares the working tree content against the index and the last commit. If
either differs, `git rm` refuses with an error like:

```text
error: the following file has local modifications:
    src/important.js
(use --cached to keep the file, or -f to force removal)
```

Read the hint: `--cached` keeps the file on disk, `-f` discards changes and
removes it from disk. Think before using `-f`.

**Removing a submodule is not the same as removing a regular directory.**
`git rm` handles submodules that use a gitfile (`.git` as a file, not a
directory) by moving the submodule's repository into the superproject's `.git`
directory and removing the entry from `.gitmodules`. Submodules that still
have a `.git` directory instead of a gitfile will not have their working tree
removed. Use `git submodule deinit` before `git rm` when in doubt.

**Files unknown to Git are silently skipped.** `git rm` only operates on
paths known to Git. If you pass an untracked file, the command exits
non-zero. This is usually what you want, but it can confuse scripts that
manage a mixed set of tracked and untracked files.

## Worked examples

### Untracking a file that was accidentally committed

A developer committed `config/database.yml` before adding it to
`.gitignore`. Other contributors have pulled the branch. The goal is to stop
tracking the file without breaking anyone's local copy.

```sh
# Verify the file is tracked
git ls-files config/database.yml

# Remove it from Git's index, keep the disk copy
git rm --cached config/database.yml

# Prevent future commits from picking it up
echo "config/database.yml" >> .gitignore
git add .gitignore

git commit -m "Stop tracking database.yml; add to .gitignore"
```

After others pull this commit, their local `config/database.yml` remains
on disk but is no longer tracked. They should ensure they have their own
local copy if they do not already.

### Cleaning up a directory of generated files

A `dist/` directory was committed by mistake. It contains dozens of
generated files across several subdirectories. Preview first, then remove.

```sh
# See what would go
git rm -n -r dist/
```

```text
rm 'dist/app.js'
rm 'dist/app.css'
rm 'dist/vendor/react.min.js'
...
```

The output looks right. Proceed:

```sh
git rm -r dist/
git commit -m "Remove dist/ from version control"
```

Add `dist/` to `.gitignore` so it stays out of the repository:

```sh
echo "dist/" >> .gitignore
git add .gitignore
git commit -m "Ignore dist/ directory"
```

### Removing files from a script idempotently

A release script cleans up old per-change log fragments after they have been
folded into the main `CHANGELOG.md`. The fragments may or may not exist on
a given run.

```sh
git rm --ignore-unmatch --quiet \
    changelogs/unreleased/*.md
```

With `--ignore-unmatch`, the script succeeds whether zero or fifty files
matched. With `--quiet`, the per-file output is suppressed so only real errors
surface.

## Recovery

If you ran `git rm` by mistake but have not yet committed, restore the file to
both the index and the working tree:

```sh
git restore --staged --worktree <path>
```

To restore an entire directory:

```sh
git restore --staged --worktree -- <directory>/
```

If you used `git rm --cached` (index-only removal), the working tree copy is
still on disk. Restore just the index entry:

```sh
git restore --staged <path>
```

If you already committed the removal, bring the file back by checking it out
from the previous commit and committing the restoration:

```sh
git checkout HEAD~1 -- <path>
git commit -m "Restore <path>"
```

See *Getting out of jams* for broader undo recipes, including recovering from
accidental force-removals.
