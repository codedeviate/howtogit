# notes

Attach, read, and manage free-form annotations on any Git object without
altering the object itself.

## Mental model

A commit is immutable once written — its hash is derived from every byte of
its content, so changing the message after the fact changes the hash. `git
notes` sidesteps that constraint by storing annotations in a separate
reference, `refs/notes/commits` by default, rather than inside the commit
object.

Think of notes as sticky tabs on the pages of a book. The text on each page
(the commit) never changes, but you can peel off and replace a sticky tab at
any time without reprinting anything.

Under the hood, notes live in their own commit history. Each time you add or
change a note, Git creates a new commit on the notes ref. The annotated
object's SHA-1 becomes a path in the notes tree; the note itself is a blob
stored at that path:

```text
Commit history        Notes ref (refs/notes/commits)
─────────────         ──────────────────────────────
  a1b2c3  ◄────────── blob: "Deployed to staging 2026-06-10"
  d4e5f6  ◄────────── blob: "CVE-2026-1234 patched here"
  ...
```

When `git log` shows a commit, it looks up the notes ref and appends any
matching note below the commit message. Notes are visible without modifying
the commit object or its hash.

## Synopsis

```text
git notes [list [<object>]]
git notes add    [-f] [--allow-empty] [-m <msg> | -F <file> | (-c|-C) <object>]
                 [-e] [--separator=<str>] [--[no-]stripspace] [<object>]
git notes append [--allow-empty] [-m <msg> | -F <file> | (-c|-C) <object>]
                 [-e] [--separator=<str>] [--[no-]stripspace] [<object>]
git notes copy   [-f] (--stdin | <from-object> [<to-object>])
git notes edit   [--allow-empty] [--[no-]stripspace] [<object>]
git notes show   [<object>]
git notes remove [--ignore-missing] [--stdin] [<object>...]
git notes prune  [-n] [-v]
git notes merge  [-v | -q] [-s <strategy>] <notes-ref>
git notes merge  --commit [-v | -q]
git notes merge  --abort  [-v | -q]
git notes get-ref
```

All subcommands default to `HEAD` when no `<object>` is given.

## Everyday usage

Add a note to the most recent commit:

```sh
git notes add -m "Deployed to staging 2026-06-10 by alice"
```

Show the note for a specific commit:

```sh
git notes show a1b2c3
```

`git log` includes notes automatically:

```sh
git log --oneline -3
```

```text
a1b2c3 Fix connection pool exhaustion
Notes:
    Deployed to staging 2026-06-10 by alice
```

Append more information to an existing note instead of overwriting it:

```sh
git notes append -m "Also deployed to production 2026-06-14" a1b2c3
```

Edit a note interactively in your configured editor:

```sh
git notes edit a1b2c3
```

Remove a note entirely:

```sh
git notes remove a1b2c3
```

List every note in the default notes ref:

```sh
git notes list
```

```text
7f3a9e... a1b2c3...
9c815d... d4e5f6...
```

## Key options

| Option | What it does | When to use it |
|--------|--------------|----------------|
| `-m <msg>` | Set the note message inline | Short, one-off annotations |
| `-F <file>` | Read the note message from a file (`-F -` reads stdin) | Long notes or notes generated by a script |
| `-f`, `--force` | Overwrite an existing note instead of aborting | Replace a note that is already set |
| `--allow-empty` | Store an empty note (default behaviour removes empty notes) | Explicitly mark a commit as reviewed with no comments |
| `-e` | Open the editor even when `-m` or `-F` is provided | Review and adjust the message before saving |
| `-C <object>` | Reuse an existing blob verbatim as the note | Copy a note from another object |
| `-c <object>` | Like `-C` but open the editor to modify it | Adapt an existing note for a new commit |
| `--separator=<str>` | Custom separator inserted between multiple `-m`/`-F` paragraphs | Use `---` instead of a blank line |
| `--no-separator` | No separator between paragraphs | Concatenate messages directly |
| `--stripspace` | Normalise whitespace (default for `-m`/`-F`) | Keep notes tidy |
| `--no-stripspace` | Preserve whitespace exactly | Notes with intentional indentation |
| `--ref=<ref>` | Operate on a notes ref other than `refs/notes/commits` | Multiple independent annotation namespaces |
| `--ignore-missing` | Do not error when removing a note that does not exist | Safe `remove` calls in scripts |
| `-n`, `--dry-run` | Show what `prune` would remove, without doing it | Audit before cleanup |
| `-s <strategy>` | Merge strategy: `manual`, `ours`, `theirs`, `union`, `cat_sort_uniq` | Control conflict resolution during `merge` |
| `--commit` | Finalise an in-progress `notes merge` after manually resolving conflicts | After editing files in `.git/NOTES_MERGE_WORKTREE` |
| `--abort` | Cancel an in-progress `notes merge` | Bail out of a conflicted merge |
| `-v` | Verbose output for `merge` (reports conflict details) and `prune` (lists removed objects) | Scripting and CI logs |
| `-q` | Quiet output for `merge` only | Suppress merge output in scripts |

## Best practices

**Use notes for information that only exists after the commit is written.**
Notes are ideal for things like deployment timestamps, downstream test
results, ticket status updates, or automated code-review scores. If the
information belongs in the commit message, write it there at commit time;
notes are not a substitute for a good message.

**Push and fetch notes explicitly — they do not travel with a normal push.**
A plain `git push` transfers your branch refs but not `refs/notes/*`. Add
the notes refspec to your remote configuration so notes travel automatically:

```sh
# One-time setup: push notes alongside branches
git config --add remote.origin.push '+refs/notes/*:refs/notes/*'

# One-time setup: fetch notes from the remote
git config --add remote.origin.fetch '+refs/notes/*:refs/notes/*'
```

Or push and fetch by hand when needed:

```sh
git push origin refs/notes/commits
git fetch origin '+refs/notes/commits:refs/notes/commits'
```

**Use separate refs for independent annotation streams.** The default
`refs/notes/commits` ref is a single shared namespace. If a CI system records
test results and a deploy script records environment data, keeping them in the
same ref clutters `git log`. Use `--ref` to separate them, and configure
`notes.displayRef` to control which namespaces each developer sees:

```sh
git notes --ref=refs/notes/ci     add -m "Build #1042: PASS" HEAD
git notes --ref=refs/notes/deploy add -m "Deployed prod 2026-06-14" HEAD

git config notes.displayRef 'refs/notes/commits'
git config --add notes.displayRef 'refs/notes/ci'
```

**Enable note rewriting so notes survive rebase and amend.** When you rebase
or amend a commit, Git creates a new commit object with a new hash. Notes
attached to the original hash become orphaned unless rewriting is configured:

```sh
git config notes.rewrite.rebase true
git config notes.rewrite.amend  true
git config notes.rewriteRef     refs/notes/commits
```

**Prefer `append` over `add -f` when notes come from multiple sources.**
Overwriting with `add -f` silently discards whatever was already there. Use
`append` to accumulate contributions from multiple people or systems without
losing history.

## Pitfalls & gotchas

**Notes are invisible to collaborators until you push them.** This is the
most common surprise. After adding notes locally and pushing the branch,
teammates see nothing. The fix is always an explicit push of
`refs/notes/commits` (or configuring the remote refspec shown above).

**Rebasing or amending silently drops notes when rewriting is not
configured.** `git rebase` and `git commit --amend` (see the *commit*
chapter) produce new objects with new hashes. Any note pointing to the old
hash becomes orphaned — it still exists in the notes ref but is never shown
because no commit carries that SHA-1 any more. Set `notes.rewrite.rebase`
and `notes.rewriteRef` before this matters.

**`add` aborts if a note already exists.** Invoking `git notes add` on an
object that already has a note exits with an error unless you pass `-f`. Use
`append` to accumulate notes, or `-f` to intentionally replace the existing
one.

**Notes history grows without bound on active repositories.** Each `add`,
`append`, `remove`, or `edit` creates a commit on the notes ref. After a
history rewrite that produces many new objects, run `git notes prune -n`
first to preview, then `git notes prune` to remove orphaned note objects.

**Merge conflicts on notes require manual intervention by default.** When two
people annotate the same commit on different clones and then merge, Git cannot
automatically reconcile the two blobs. The merge pauses, checks out the
conflicting notes in `.git/NOTES_MERGE_WORKTREE`, and waits. Edit those files,
then run `git notes merge --commit` to finalise, or `git notes merge --abort`
to bail out. Alternatively, choose an automated strategy such as `union` or
`cat_sort_uniq` upfront.

**The `--ref` value is normalised, not passed verbatim.** A value starting
with `refs/notes/` is used as-is; one starting with `notes/` gets `refs/`
prepended; anything else gets `refs/notes/` prepended. So `--ref=ci` targets
`refs/notes/ci`, not `refs/ci`. Verify with `git notes get-ref` after setting
`GIT_NOTES_REF` or `core.notesRef`.

## Worked examples

### Recording deployment events without modifying commits

A deploy script annotates each commit it ships so that the release history is
visible in `git log` without any secondary database.

```sh
# In the deploy pipeline
COMMIT=$(git rev-parse HEAD)
TIMESTAMP=$(date -u +%Y-%m-%dT%H:%M:%SZ)
git notes --ref=refs/notes/deploy append \
    -m "Deployed to production at ${TIMESTAMP} by ${DEPLOY_USER}" \
    "$COMMIT"
git push origin refs/notes/deploy
```

A developer fetches and enables the namespace:

```sh
git fetch origin '+refs/notes/deploy:refs/notes/deploy'
git config --add notes.displayRef refs/notes/deploy
git log --oneline -3
```

```text
a1b2c3 Fix connection pool exhaustion
Notes (refs/notes/deploy):
    Deployed to production at 2026-06-14T09:41:00Z by ci-bot
d4e5f6 Increase pool size to 20
Notes (refs/notes/deploy):
    Deployed to production at 2026-06-10T14:22:00Z by ci-bot
```

No commits were altered; the deployment record lives entirely in the notes
ref.

### Adding a post-hoc code-review note to a shared branch

You have reviewed a colleague's branch and want to record the outcome against
the specific commit you reviewed, not as a comment in the pull-request
interface.

```sh
# Identify the commit
git log --oneline feature/payment-refactor
# b7d91a Refactor Stripe checkout path
# 3cc4f0 Extract PaymentGateway interface

# Write the review note
git notes add -m "Code review by alice 2026-06-17: approved.
Latency concern tracked in #892." b7d91a

# Share it
git push origin refs/notes/commits
```

A teammate retrieves and reads it:

```sh
git fetch origin '+refs/notes/commits:refs/notes/commits'
git notes show b7d91a
```

```text
Code review by alice 2026-06-17: approved.
Latency concern tracked in #892.
```

### Resolving a notes merge conflict

Two CI systems both write notes to the same commit on the same notes ref. When
you merge the two notes branches a conflict arises.

```sh
git notes merge refs/notes/other-ci
# CONFLICT (content): Merge conflict in bf/fe30.../680d5a...
# Automatic merge failed; fix conflicts and then commit the result.
```

Inspect and edit the conflicting file in the worktree:

```sh
ls .git/NOTES_MERGE_WORKTREE/
vim .git/NOTES_MERGE_WORKTREE/bf/fe30.../680d5a...
# Remove conflict markers, keep both entries
git notes merge --commit
```

Or restart the merge using the `union` strategy to concatenate automatically:

```sh
git notes merge --abort
git notes merge -s union refs/notes/other-ci
```

## Recovery

Remove a note you added by mistake:

```sh
git notes remove HEAD
```

If you need to recover a note that was deleted, the notes ref has its own
commit history. Inspect it to find the state just before the deletion:

```sh
git log --oneline refs/notes/commits
# Find the commit just before the delete
git show <old-notes-commit>:<first-2-chars-of-sha>/<rest-of-sha>
```

If a rebase or amend silently dropped notes because `notes.rewrite` was not
configured, retrieve the old note using the original commit SHA from the
reflog:

```sh
# Find the original commit SHA
git reflog

# Read the note from the historical notes tree
git log --all --oneline refs/notes/commits
git show <historical-notes-commit>:<path-in-notes-tree>
```

See *Getting out of jams* for broader strategies around recovering lost
objects.
