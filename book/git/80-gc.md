# gc

Clean up unnecessary files and optimize the local repository by packing loose
objects, pruning unreachable history, and updating ancillary indexes.

## Mental model

Every time you commit, amend, rebase, or stash, Git writes new objects to the
repository's object database — blobs for file contents, trees for directories,
and commit objects tying them together. Some of those objects eventually become
unreachable: the dangling commit left behind by an amend, the blob that was
staged but never committed, the stash entry you dropped. They take up disk
space and slow object lookups.

Git also stores objects in two ways. Small, recently created objects live as
individual files under `.git/objects/` — one file per object, named by its
SHA-1 hash (loose objects). Over time, `git gc` gathers loose objects into a
single binary file called a packfile, which applies delta compression between
similar objects and is far more space-efficient and faster to read.

Think of `git gc` as a janitor that does four things in one pass:

1. **Repacks** loose objects into a packfile, delta-compressing the contents.
2. **Prunes** unreachable loose objects that are older than the grace period
   (default: two weeks).
3. **Packs refs** from individual files under `.git/refs/` into `.git/packed-refs`.
4. **Updates ancillary indexes** — the commit-graph, the reflog expiry, and
   rerere metadata.

You rarely need to run `git gc` by hand. Porcelain commands (`git commit`,
`git fetch`, `git pull`) call `git gc --auto` automatically once the number of
loose objects or packs crosses a configurable threshold.

```text
.git/objects/xx/yyyy...  (loose objects, one per SHA)
           │
           └─ git gc ─> .git/objects/pack/pack-<sha>.{idx,pack}
                                           (delta-compressed packfile)

Unreachable objects older than gc.pruneExpire (default 2 weeks) → deleted
```

## Synopsis

```text
git gc [--aggressive] [--auto] [--[no-]detach] [--quiet]
       [--prune=<date> | --no-prune] [--force] [--keep-largest-pack]
```

## Everyday usage

Run a standard housekeeping pass:

```sh
git gc
```

Output looks like:

```text
Enumerating objects: 4021, done.
Counting objects: 100% (4021/4021), done.
Delta compression using up to 10 threads
Compressing objects: 100% (1387/1387), done.
Writing objects: 100% (4021/4021), done.
Total 4021 (delta 2614), reused 4021 (delta 2614), pack-reused 0
```

Check whether housekeeping is needed before doing any work (the same check
automatic gc performs):

```sh
git gc --auto
```

Prune all unreachable objects immediately rather than waiting for the default
two-week grace period:

```sh
git gc --prune=now
```

Suppress progress output in scripts or cron jobs:

```sh
git gc --quiet
```

Run in the foreground and wait for completion, even if background detach is
configured:

```sh
git gc --no-detach
```

## Key options

| Option | What it does | When to use it |
|---|---|---|
| `--aggressive` | Repack with `--no-reuse-delta`, larger window/depth; takes much longer | One-off optimization of a repo with a history of many force-pushes or filter-branch runs |
| `--auto` | Exit immediately if housekeeping thresholds are not exceeded | Called by porcelain commands; rarely useful interactively |
| `--[no-]detach` | Override the `gc.autoDetach` config to run in foreground or background | Use `--no-detach` in scripts that must wait for gc to finish |
| `--prune=<date>` | Delete unreachable objects older than `<date>`; `--prune=now` deletes all | Safe on a single-user repo; risky when other processes may be writing |
| `--no-prune` | Skip deletion of unreachable loose objects entirely | Skip pruning when you want to inspect unreachable objects first with `git fsck --unreachable` |
| `--force` | Run even if another `git gc` may already be running | Use with care; two concurrent gc runs can corrupt packs |
| `--keep-largest-pack` | Consolidate all packs except the largest non-cruft pack, any `.keep`-marked packs, and cruft packs | Reduce pack count while preserving the pack with the most data |
| `--quiet` | Suppress progress output | CI, cron, and scripts where noise is unwanted |

## Best practices

**Let automatic gc do its job.** The default `gc.auto` threshold (6700 loose
objects) and `gc.autoPackLimit` threshold (50 packs) are well-tuned for most
workflows. Manual runs are needed only after bulk imports, after heavy use of
`git filter-branch`, or when you want to reclaim space on a constrained
machine.

**Do not use `--prune=now` on shared repositories.** The two-week grace period
exists because `git gc` can run concurrently with another process that has
written objects but has not yet created references to them. Pruning those
objects immediately can corrupt the repository for the other process.

**Treat `--aggressive` as a one-off, not a habit.** It throws away all
existing deltas and recomputes them from scratch using a larger window (250
by default vs. 10 normally). The improvement is mostly persistent, so running
it once after a major history rewrite is worthwhile; running it repeatedly on
a healthy repo wastes time without meaningful gain.

**Schedule `git gc --quiet` in maintenance windows for server-side repos.**
Hosting platforms run their own maintenance, but self-hosted bare repositories
benefit from a weekly cron job. Pair `--quiet` with output redirection so
failures are noticed without noise on success. Consider the newer
`git maintenance` command (see the *maintenance* chapter) for finer-grained
scheduling.

**Use `gc.bigPackThreshold` to protect large packs from being repacked.**
Set it to the size of your baseline pack to avoid unpacking and repacking
hundreds of megabytes every time small changesets arrive:

```sh
git config gc.bigPackThreshold 200m
```

## Pitfalls & gotchas

**`git gc` will not delete objects still referenced anywhere.** References
include not just branches and tags but also the index, remote-tracking
branches, reflogs, and everything under `refs/*`. If you expect an object to
be deleted but it persists, check whether a reflog, stash, or remote-tracking
branch still points to it.

**The two-week grace period can mask expected pruning.** After a rebase or
amend, the old commits remain as loose objects until `gc.pruneExpire` elapses.
Running `git gc` the day after does not remove them. Run `git gc --prune=now`
if you explicitly want them gone immediately (single-user repos only).

**Background detach by default can hide failures.** With `gc.autoDetach=true`
(the default), `git gc --auto` returns immediately. Errors are written to
`.git/gc.log` and suppressed for `gc.logExpiry` (default: one day) to avoid
spamming every subsequent command. Check `.git/gc.log` if you suspect gc is
failing silently.

**`--force` is not safe to use routinely.** If a gc run is truly in progress
in another process (not just a stale lock file), forcing a second concurrent
run can leave packs in an inconsistent state. Verify the other process is dead
before using `--force`.

**`--aggressive` may produce larger packfiles in some cases.** The repack
discards existing deltas and recomputes them. Occasionally the new computation
finds a worse set of deltas for certain object graphs, resulting in a slightly
larger pack. This is acknowledged in the documentation and is an accepted
trade-off.

## Worked examples

### Reclaiming space after rewriting history

You have just run `git filter-branch` (or `git filter-repo`) to remove a large
binary that was accidentally committed. The rewrite created thousands of new
objects and left the old ones dangling.

```sh
# Confirm there are many loose objects before gc
git count-objects -v
```

```text
count: 8342
size: 94832
in-pack: 0
packs: 1
size-pack: 12451
prune-packable: 0
garbage: 0
size-garbage: 0
```

```sh
# Remove the old refs that filter-branch left in refs/original/
git for-each-ref --format="%(refname)" refs/original/ \
  | xargs -r git update-ref -d

# Expire all reflog entries that still reference the old objects
git reflog expire --expire=now --all

# Prune immediately and repack aggressively
git gc --aggressive --prune=now
```

```sh
# Verify the reduction
git count-objects -v
```

```text
count: 0
size: 0
in-pack: 3105
packs: 1
size-pack: 4218
prune-packable: 0
garbage: 0
size-garbage: 0
```

The repository has shrunk from ~95 MB of loose objects plus a 12 MB pack down
to a single 4 MB pack.

### Running gc in a CI maintenance job

A self-hosted bare repository is used as the central remote for a team.
You want a weekly cron job that compacts the repository without interrupting
pushes or fetches.

```sh
# /etc/cron.weekly/git-gc-repos
#!/bin/sh
set -e
REPOS_ROOT=/srv/git
LOG=/var/log/git-gc.log

find "$REPOS_ROOT" -name HEAD -execdir \
  git gc --quiet --no-detach \; >> "$LOG" 2>&1
```

`--no-detach` ensures the script waits for gc to finish before the cron job
exits, so the log captures the full output and errors surface immediately.
`--quiet` suppresses normal progress output so only errors appear in the log.

### Inspecting what would be pruned

Before committing to a destructive `--prune=now`, audit the unreachable
objects:

```sh
# List all unreachable objects
git fsck --unreachable 2>/dev/null | grep "^unreachable commit" | awk '{print $3}'
```

Pipe one of those hashes to `git show` to confirm it is safe to remove:

```sh
git show <hash>
```

If the output shows a commit or file you no longer need, proceed with:

```sh
git gc --prune=now
```

See the *fsck* chapter for a deeper look at inspecting repository integrity.

## Recovery

`git gc` deletes objects permanently once they have been pruned. If you ran
`--prune=now` and removed commits you still needed, recovery depends on
whether those commits are recorded in the reflog.

Check the reflog first:

```sh
git reflog
```

If the commit hash appears there, restore the branch:

```sh
git branch recover-branch <hash>
```

If the reflog does not contain it (for example, the reflog was also expired),
and you did not push the branch anywhere, the objects are unrecoverable. This
is why the two-week default grace period exists and why `--prune=now` should
be used only when you are certain.

See *Getting out of jams* for reflog-based recovery recipes.
