# mv

Move or rename a file, directory, or symlink and immediately stage the result.

## Mental model

When you rename a file outside of Git — with your OS file manager or the shell
`mv` command — Git sees two unrelated events: the old path was deleted and a
new, unrelated path was created. It cannot know they are the same file. `git
status` shows one deletion and one untracked addition, and the index records
them as two separate changes. Worse, tools that display rename detection (like
`git log --follow`) have to guess the relationship later from content
similarity.

`git mv` solves this by combining the filesystem rename with an index update in
one step:

1. Renames the file on disk (the same operation your shell `mv` would do).
2. Stages the deletion of the old path.
3. Stages the addition of the new path.

After `git mv`, `git status` shows the change as a single `renamed:` entry and
the index already reflects the intended final state. You still need to
`git commit` to record the rename in history.

```text
Before:                           After git mv old.txt new.txt:
  working tree: old.txt             working tree: new.txt
  index:        old.txt             index:        old.txt -> new.txt (staged rename)
  HEAD:         old.txt             HEAD:         old.txt  (unchanged until commit)
```

Git does not actually store rename events in its object model — it reconstructs
rename history from content similarity at display time. `git mv` just makes
that reconstruction reliable by ensuring the old and new paths appear together
in the same commit with identical content.

## Synopsis

```text
git mv [-v] [-f] [-n] [-k] <source> <destination>
git mv [-v] [-f] [-n] [-k] <source>... <destination-directory>
```

In the first form, `<source>` is renamed to `<destination>`. In the second
form, `<destination-directory>` must already exist and each `<source>` is
moved into it.

## Everyday usage

Rename a single file:

```sh
git mv README.txt README.md
git commit -m "Rename README to Markdown"
```

Move a file into an existing subdirectory:

```sh
git mv logo.png assets/images/logo.png
git commit -m "Move logo into assets directory"
```

Move several files into a directory at once:

```sh
git mv auth.js session.js token.js src/auth/
git commit -m "Gather authentication modules under src/auth"
```

Rename a directory:

```sh
git mv lib/ src/
git commit -m "Rename lib to src"
```

Preview a move without touching anything:

```sh
git mv -n old-name.js new-name.js
```

```text
Checking rename of 'old-name.js' to 'new-name.js'
Renaming old-name.js to new-name.js
```

## Key options

| Option | What it does | When to use it |
|--------|--------------|----------------|
| `-f`, `--force` | Overwrite `<destination>` if it already exists | Replace a tracked file with a renamed one |
| `-k` | Skip sources that would cause an error rather than aborting | Batch moves where some sources may not exist or may not be tracked |
| `-n`, `--dry-run` | Show what would happen without making any changes | Verify paths before a destructive rename |
| `-v`, `--verbose` | Print each file name as it is moved | Confirm progress during a large batch move |

## Best practices

**Commit renames in their own commit, separate from content edits.** When a
rename and a content change land in the same commit, `git diff` and code review
tools have to do extra work to distinguish intent from implementation. A
reviewer seeing `src/util.js` renamed to `src/helpers/format.js` plus a
content change may miss a subtle bug. Put the rename commit first, then follow
with the content change.

**Check the destination does not already exist before moving.** If
`<destination>` is a tracked file and you run `git mv` without `-f`, Git
refuses. Using `-f` silently discards the destination file. Use `git mv -n`
first to confirm the target path is free.

**Use `git mv` rather than shell `mv` plus manual staging.** You can reach the
same end state with `mv old.txt new.txt && git rm old.txt && git add new.txt`,
but `git mv` is less error-prone and communicates intent clearly in your shell
history. It also handles symlinks and the `.gitmodules` update for submodules
automatically.

**For large restructures, move in small batches.** Moving a hundred files in
one commit is hard to review. Break the restructure into logical groups — for
example, one commit per top-level directory — so reviewers can follow the
reasoning.

**After moving a submodule, run `git submodule update`.** Git updates
`.gitmodules` and the submodule's `core.worktree` setting automatically, but
does not repopulate the new location. Run `git submodule update` to check out
the submodule content at the new path.

## Pitfalls & gotchas

**Moving an untracked file fails.** `git mv` only operates on paths that are
already tracked. If a file has never been staged, Git treats it as non-existent
and refuses to move it. Add it first, then move it:

```sh
git add newfile.txt
git mv newfile.txt dest/newfile.txt
```

Alternatively, move it with the shell and `git add` the new path — since it was
never tracked, there is no old index entry to clean up.

**The destination directory must already exist.** `git mv` will not create
intermediate directories. If `src/auth/` does not exist yet, the move fails.
Create it first:

```sh
mkdir -p src/auth
git mv auth.js src/auth/
```

Git does not track empty directories, so you may want to add a `.gitkeep` file
if the directory needs to be preserved before other files arrive.

**`-k` silently skips errors — verify the result.** In a batch operation with
`-k`, any source that is untracked or would overwrite an existing file is
silently skipped rather than aborting. Always run `git status` after a `-k`
batch to confirm every intended move happened.

**Rename detection has limits.** Git detects renames at display time when
content similarity between the old and new path exceeds a threshold (50% by
default). If you rename a file and make extensive edits in the same commit, Git
may fail to recognise it as a rename in `git log --follow` or `git diff`.
Keeping the rename commit free of content edits avoids this.

**Case-only renames fail silently on case-insensitive filesystems.** On macOS
and Windows, renaming `Config.js` to `config.js` may appear to succeed but the
OS never actually changes the filename on disk. Use a two-step rename through a
temporary name:

```sh
git mv Config.js config.js.tmp
git mv config.js.tmp config.js
git commit -m "Lowercase Config.js filename"
```

## Worked examples

### Reorganising a flat source tree into subdirectories

All source files live in the project root. You want to move them into `src/`.

```sh
mkdir src
git mv app.js utils.js router.js src/
git status
```

```text
On branch main
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
	renamed:    app.js -> src/app.js
	renamed:    router.js -> src/router.js
	renamed:    utils.js -> src/utils.js
```

```sh
git commit -m "Move source files into src directory"
```

Later, `git log --follow src/app.js` traces history back through the rename to
the original `app.js` commits.

### Renaming a file with a case-only change on a case-insensitive filesystem

The file is currently `UserModel.js` but the team convention is lowercase
filenames. A direct rename will not work on macOS or Windows.

```sh
# Step 1: rename to a temporary name
git mv UserModel.js usermodel_tmp.js

# Step 2: rename from temp to the final lowercase name
git mv usermodel_tmp.js usermodel.js

git status
```

```text
On branch main
Changes to be committed:
	renamed:    UserModel.js -> usermodel.js
```

```sh
git commit -m "Rename UserModel.js to usermodel.js (enforce lowercase convention)"
```

### Recovering when you already used shell mv

You ran `mv src/old.js src/new.js` in the shell before thinking about Git.
The index now shows a deletion and an untracked file. Fix it:

```sh
git rm src/old.js
git add src/new.js
git status
```

```text
On branch main
Changes to be committed:
	renamed:    src/old.js -> src/new.js
```

Git's rename detection recognises the content match and promotes the pair to a
`renamed:` entry. The effect is identical to having used `git mv` from the
start. Commit normally.

## Recovery

To undo a staged rename before committing, move the file back:

```sh
git mv src/new.js src/old.js
```

Or use *restore* to unstage both sides and return the working tree to the state
at HEAD:

```sh
git restore --staged src/old.js src/new.js
rm src/new.js             # remove the new path from the working tree
git restore src/old.js    # src/old.js is back in place
```

If you have already committed a rename and want to reverse it, use *revert* to
create a new commit that undoes it (see the *revert* chapter). If the commit
exists only on your local machine, `git reset HEAD~1` brings it back to the
staging area so you can re-rename before recommitting (see the *reset* chapter).

See *Getting out of jams* for broader undo recipes involving staged changes.

## See also

- *add* — staging changes manually when you have already renamed with the shell.
- *rm* — the counterpart for removing tracked files from the index.
- *status* — verifying that renames appear as `renamed:` rather than separate
  delete/add pairs.
- *commit* — recording the staged rename into history.
- *restore* — unstaging a rename without losing working-tree changes.
- *reset* — rolling back a rename commit that has not yet been pushed.
- *revert* — safely undoing a rename that has already been shared.
