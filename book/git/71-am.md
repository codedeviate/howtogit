# am

Apply a series of patches from a mailbox file, reconstructing commits with
their original author, date, and message.

## Mental model

`git am` is the receiving end of an email-based patch workflow. The sending
end — `git format-patch` — turns commits into RFC 2822 email messages where
the subject line becomes the commit title, the body becomes the commit message,
and the diff is appended inline. `git am` reverses that process: it reads one
or more of those messages, extracts the author identity and date from the email
headers, applies the diff to the working tree, and records a real commit.

```text
Contributor machine                     Maintainer machine
─────────────────                       ──────────────────
git format-patch origin/main            git am 0001-fix-login.patch
  └─> 0001-fix-login.patch  ──email──>    └─> commit in history
```

Think of it as `git apply` plus `git commit` in one step, with the commit
metadata coming from the email message rather than from you interactively.
Because the original author and date are preserved, the contributor's identity
appears in `git log` even though you are the one running the command on your
machine.

`git am` is stateful: when a patch fails to apply, it pauses and leaves a
session in `.git/rebase-apply/`. You resolve the conflict, then tell it to
`--continue`, `--skip`, or `--abort`, just like a rebase. The command refuses
to start a new session until the current one is resolved or aborted.

## Synopsis

```text
git am [--signoff] [--keep] [--[no-]keep-cr] [--[no-]utf8]
       [--[no-]3way] [--interactive] [--no-verify]
       [--committer-date-is-author-date] [--ignore-date]
       [--ignore-space-change | --ignore-whitespace]
       [--whitespace=<action>] [-C<n>] [-p<n>]
       [--directory=<dir>] [--exclude=<path>] [--include=<path>]
       [--reject] [-q | --quiet] [--[no-]scissors]
       [-S[<keyid>]] [--patch-format=<format>]
       [--quoted-cr=<action>] [--empty=(stop|drop|keep)]
       [(<mbox> | <Maildir>)...]

git am (--continue | --skip | --abort | --quit | --retry |
        --show-current-patch[=(diff|raw)] | --allow-empty)
```

## Everyday usage

Apply a single patch file produced by `git format-patch`:

```sh
git am 0001-fix-null-pointer-in-login.patch
```

Apply all patches in the current directory in order:

```sh
git am *.patch
```

Apply patches piped from standard input (common when saving email directly):

```sh
git am < incoming.mbox
```

Apply a patch with 3-way merge fallback so conflicts land in the working tree
rather than causing a hard failure:

```sh
git am --3way 0001-fix-null-pointer-in-login.patch
```

Add your own `Signed-off-by` trailer to each applied commit (required by some
projects such as the Linux kernel):

```sh
git am --signoff patches/*.patch
```

## Key options

| Option | What it does | When to use it |
|--------|--------------|----------------|
| `-3`, `--3way` | Fall back to 3-way merge when the patch does not apply cleanly | Almost always — makes conflicts resolvable instead of fatal |
| `-s`, `--signoff` | Append a `Signed-off-by` trailer with your identity | Projects that require DCO sign-off from the person applying the patch |
| `--keep` | Pass `-k` to `git mailinfo`; preserve the original subject prefix | Prevent `[PATCH]` or `[RFC]` from being stripped out of the title |
| `--no-utf8` | Do not re-encode the commit message to UTF-8 | Rare; when the source encoding must be preserved verbatim |
| `--[no-]keep-cr` | Preserve (or strip) carriage returns at line endings | Patches originating from Windows mail clients |
| `--scissors` | Drop everything before the scissors line (`-- >8 --`) in the email body | Patches forwarded with extra quoting above the actual diff |
| `-p<n>` | Strip `<n>` leading path components from filenames in the diff | Applying a patch with a different directory depth |
| `-C<n>` | Require `<n>` lines of surrounding context to match | Tighten the match when the patch was made against a different base |
| `--directory=<dir>` | Prepend `<dir>` to all filenames in the patch | Applying a patch into a subdirectory |
| `--reject` | Write unmatched hunks as `.rej` files instead of aborting | Partial application when you want to salvage what applies cleanly |
| `--whitespace=<action>` | Control how whitespace errors are handled (`nowarn`, `warn`, `fix`, `error`, `error-all`) | `fix` automatically corrects trailing whitespace issues |
| `--patch-format=<fmt>` | Force a specific patch format (`mbox`, `mboxrd`, `stgit`, `stgit-series`, `hg`) | When auto-detection picks the wrong format |
| `--empty=(stop\|drop\|keep)` | What to do with email messages that contain no patch | `drop` skips non-patch messages silently; default is `stop` |
| `-m`, `--message-id` | Include the `Message-ID` header in the commit message | Traceability back to the mailing list archive |
| `--committer-date-is-author-date` | Set the committer date to match the author date from the email | Reproducible builds or bisect scenarios requiring exact timestamps |
| `--ignore-date` | Ignore the email date; use the current time as author date | When the patch is old but you want today's timestamp |
| `-q`, `--quiet` | Print only error messages | Scripted bulk application |
| `-S[<keyid>]`, `--gpg-sign` | GPG-sign each applied commit | Verified commit workflows |
| `--no-verify` | Skip the `applypatch-msg` and `pre-applypatch` hooks | Emergency; investigate the failing hook afterward |
| `--interactive` | Confirm each patch before applying | Reviewing a batch of patches one at a time |
| `--continue`, `-r`, `--resolved` | Resume after manually resolving a conflict | Conflict recovery (see Recovery section) |
| `--skip` | Discard the current failing patch and move to the next | When a patch is genuinely inapplicable and you want to move on |
| `--abort` | Restore the branch to its state before `git am` started | Start over from scratch |
| `--quit` | Stop the session without restoring HEAD or the index | Rare; leaves the tree in mid-session state |
| `--retry` | Retry the last failing patch, useful for passing extra options | Re-attempt with `--3way` after an initial failure without it |
| `--show-current-patch[=(diff\|raw)]` | Print the patch that caused the current failure | Inspect what you are trying to resolve |

## Best practices

**Always pass `--3way` for patches against a moving target.** A patch
produced against an old commit rarely applies perfectly by line number. With
`--3way`, git falls back to a three-way merge when the context does not match
exactly, giving you standard conflict markers in the working tree instead of a
hard failure. Make it your default by setting `am.threeWay` in your config:

```sh
git config --global am.threeWay true
```

**Verify the patch before applying it.** `git am` creates real commits in your
history. Inspect the patch file first to understand what it changes:

```sh
git apply --stat 0001-fix-login.patch   # summary of changed files
git apply --check 0001-fix-login.patch  # dry-run: does it apply cleanly?
```

If `--check` passes, `git am` will too (absent 3-way considerations).

**Keep patches numbered and ordered.** `git format-patch` names files
`0001-…`, `0002-…`. Apply them in that order. Applying `0003` before `0002`
almost always fails because later patches depend on earlier ones. Use a glob
that expands in numeric order:

```sh
git am *.patch
```

**Confirm the applied commits before pushing.** After a successful `git am`,
run `git log --oneline` to confirm author names, dates, and messages look
correct. A misformatted patch can corrupt the author line silently:

```sh
git log --format="%h %aN <%aE> %s" -5
```

**Use `--signoff` when the project requires it.** The Linux kernel, U-Boot,
and many other projects treat the `Signed-off-by` trailer as a legal
declaration under the Developer Certificate of Origin. If you are the person
integrating a patch on behalf of a contributor, add your own sign-off:

```sh
git am --signoff patches/*.patch
```

This appends `Signed-off-by: Your Name <you@example.com>` to every commit
while preserving the contributor's original authorship.

## Pitfalls & gotchas

**Patch context lines must match the working tree.** `git am` computes patch
offsets using surrounding context. If the base commit differs too much from
what the patch was created against, even `--3way` may not help. Check whether
the patch records base commit information (produced by `git format-patch
--base=...`) and, if possible, apply onto the same base first.

**Email clients mangle patches.** Quoted-printable encoding, line-wrapping at
76 characters, converting tabs to spaces, and HTML multipart messages all
corrupt patches in transit. The safest delivery method is attaching the raw
`.patch` file; the second safest is saving the raw message from a command-line
mail client with no re-encoding. If the patch arrives mangled, `--whitespace=fix`
and `--keep-cr` may help at the margins, but there is no substitute for a clean
raw file. The `git send-email` / `git am` pair avoids this problem entirely by
sending patches as plain-text attachments.

**`git am` is not the same as `git apply`.** `git apply` modifies the working
tree and optionally the index but creates no commit. `git am` always creates a
commit. If you want to inspect or partially apply a change without committing,
use the *apply* command instead.

**ORIG_HEAD is set before any patches are applied.** Git records the tip of
the branch in `ORIG_HEAD` at the start of the session. If something goes wrong
mid-series you can return to the pre-am state:

```sh
git reset --hard ORIG_HEAD
```

This is faster than `--abort` after a failed session and works even if the
session state has been corrupted.

**An in-progress session blocks new work.** If you run `git am` while a
previous session is unfinished, git refuses:

```text
error: previous rebase directory .git/rebase-apply still exists but mbox given.
```

Either finish the session (`--continue`, `--skip`) or abandon it (`--abort`)
before starting a new one.

**Empty messages stop the session by default.** If an mbox contains forwarded
emails or cover letters without a diff, `git am` halts with `--empty=stop`
(the default). Pass `--empty=drop` to silently skip non-patch messages:

```sh
git am --empty=drop patches.mbox
```

## Worked examples

### Applying a patch series from a mailing list

A contributor sends a three-patch bug-fix series. You save the thread as an
mbox file from your mail client (`patches.mbox`).

```sh
# Preview what will change
git apply --stat patches.mbox

# Dry-run to catch obvious failures early
git apply --check patches.mbox

# Apply for real, with 3-way fallback and your sign-off
git am --3way --signoff patches.mbox
```

Successful output:

```text
Applying: Fix null pointer in login handler
Applying: Add regression test for login edge case
Applying: Update CHANGELOG for login fix
```

Verify the result:

```sh
git log --oneline -3
```

```text
c4a1e8f Update CHANGELOG for login fix
9d73b2a Add regression test for login edge case
3f51a04 Fix null pointer in login handler
```

### Resolving a conflict mid-series

You are applying a five-patch series. Patch 3 fails:

```text
Applying: Refactor authentication module
error: patch failed: src/auth.c:142
error: src/auth.c: patch does not apply
hint: Use --reject to leave rejected hunks in files with .rej extension.
hint: Alternatively, use git am --skip to skip this patch.
hint: To restore the original branch and stop patching run "git am --abort".
```

Inspect the failing patch:

```sh
git am --show-current-patch=diff
```

If you did not start with `--3way`, abort and retry with it:

```sh
git am --abort
git am --3way patches/*.patch
```

If conflict markers appear in the working tree, resolve them manually, stage
the result, and continue:

```sh
# Edit src/auth.c to resolve the conflict markers
git add src/auth.c
git am --continue
```

`git am` applies the remaining patches automatically. If patch 4 or 5 also
conflicts, repeat the process for each one.

### Applying a patch from a GitHub commit URL

GitHub's patch endpoint (`/commit/<sha>.patch`) returns a valid mbox message.
Fetch and apply it directly:

```sh
curl -L https://github.com/example/repo/commit/abc1234.patch | git am --3way
```

Successful output:

```text
Applying: Fix race condition in worker pool
```

If it fails, inspect the situation with `--show-current-patch`, resolve, and
run `git am --continue`.

### Recovering from applying onto the wrong branch

You realize mid-session that you started on the wrong branch. Abort completely:

```sh
git am --abort
git switch correct-branch
git am --3way patches/*.patch
```

`--abort` restores the branch to the state captured in `ORIG_HEAD` before any
patches were applied.

## Recovery

If patches have been applied and you want to undo all of them at once:

```sh
git reset --hard ORIG_HEAD
```

`ORIG_HEAD` is set automatically when `git am` starts. This is the fastest
path when the session completed but the result is wrong.

To abort a session that is currently paused on a conflict:

```sh
git am --abort
```

This restores the branch and clears the `.git/rebase-apply/` state directory.

To undo only the most recently applied commit after a clean `git am`:

```sh
git reset --soft HEAD~1    # keep changes staged
# or
git reset HEAD~1           # keep changes in the working tree
```

To see all commits applied in the session (useful before deciding how far to
reset):

```sh
git reflog
```

See *Getting out of jams* for additional undo recipes and how to recover
commits using the *reflog* command.
